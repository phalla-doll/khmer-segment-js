## 2. MUST-Fix / Explicitly-Accept Risks

### 2.1 `groupDigitTokens` type coupling (`FmmToken[]` vs `SegmentToken[]`)

**Files:** `src/algorithms/group-digits.ts`, `src/core/segment.ts`

`groupDigitTokens` is typed as `FmmToken[]`, but `segmentWords` passes `SegmentToken[]` and casts the result back. It works today due to structural compatibility, but this creates hidden coupling between unrelated algorithm types.

**Action:** Change `groupDigitTokens` signature to `SegmentToken[]` from `src/types/public.ts` and remove cast in `segmentWords`.

### 2.2 `deleteBackward(..., { normalize: true })` cursor-space contract ambiguity

**Files:** `src/core/caret.ts`, `src/__tests__/caret.test.ts`

When normalization changes text length (e.g. removes ZWS), `cursorIndex` is clamped against normalized text, not original text. This behavior may be acceptable, but the contract is not tested/documented clearly enough for editor integrations.

**Action:** Add explicit tests with normalization-changing input and document index-space behavior in README/API docs.

### 2.3 `Trie.hasSuffix()` implementation is expensive and not used by current dictionary path

**Files:** `src/dictionary/trie.ts`, `src/dictionary/memory-dictionary.ts`

`Trie.hasSuffix()` performs deep recursive search and can be exponential. `MemoryDictionary.hasSuffix()` does not use this path (it uses reverse trie + `hasPrefix`), so this method is effectively unused by current flow and risky if reused later.

**Action:** Either remove `Trie.hasSuffix()` or replace with linear/prefix-based suffix strategy.

**Important:** Keep `KhmerDictionary.hasSuffix?` interface support; only adjust internal trie method.

## 3. Important Improvements (Not Release Blockers)

### 3.1 Duplicate helpers increase maintenance risk

- `isRobat()` duplicated in `src/core/cluster.ts`, `src/core/normalize.ts`, `src/algorithms/viterbi.ts`
- `cpAt()` duplicated in `src/core/detect.ts`, `src/core/cluster.ts`, `src/core/normalize.ts`, `src/algorithms/viterbi.ts`

**Action:** Centralize in `src/constants/char-categories.ts` (or a shared util).

### 3.2 Viterbi cluster walking logic is partially duplicated

**File:** `src/algorithms/viterbi.ts`

`getClusterLength()` mirrors cluster continuation logic from `splitClusters`-related rules. Divergence risk exists if cluster rules change later.

**Action:** Extract a shared `walkCluster()` helper or consume cluster boundaries from shared logic.

### 3.3 Lint warnings remain in repository

Warnings are mostly in tests, but there are production warnings in `src/dictionary/trie.ts` (`no-non-null-assertion`).

**Action:** Decide whether to (a) clean warnings or (b) codify acceptable warning policy.

## 4. Performance and Distribution Notes

### 4.1 Dictionary bundle remains large (expected for data-heavy module)

Current measured output:

- `dist/dictionary/index.cjs`: ~8.08 MB
- `dist/dictionary/index.js`: ~8.08 MB
- sourcemaps: ~23 MB combined

This is acceptable for opt-in `khmer-segment/dictionary`, but still heavy for browser delivery if loaded eagerly.

**Action:** Consider optional lazy loading / externalized JSON in a future version.

### 4.2 Perf tests do extra segmentation pass for assertions

**File:** `src/__tests__/segment-perf.test.ts`

Each case benchmarks `segmentWords(...)`, then calls it again for correctness assertions.

**Action:** Return the final run output from benchmark helper or split benchmark and correctness concerns.
