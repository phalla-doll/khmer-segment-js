## Part 2: v0.4.0 — Caret/Text-Editing APIs + Viterbi Decision

### 2.1 `getCaretBoundaries(text, options?)`

**File**: New `src/core/caret.ts`

Returns an array of valid caret positions (indices where the cursor can rest) based on Khmer cluster boundaries. Uses existing `splitClusters()` internally to find safe break points.

```ts
interface CaretOptions {
    /** Whether to normalize the text before computing boundaries. Default: false. */
    normalize?: boolean;
}

function getCaretBoundaries(text: string, options?: CaretOptions): number[];
```

**Behavior**:

- Returns a sorted array of valid caret positions.
- Each cluster boundary becomes a caret stop.
- Position `0` (before all text) and position `text.length` (after all text) are always included.
- Default behavior (`normalize: false`) computes boundaries directly on the input string so indices stay in the caller's original text space.
- When `normalize: true`, operates on normalized text; returned positions are in normalized-text index space.

**Examples**:

- `getCaretBoundaries("")` → `[0]`
- `getCaretBoundaries("ក")` → `[0, 1]`
- `getCaretBoundaries("ក្ក")` → `[0, 2]` (coeng+subscript is one cluster)
- `getCaretBoundaries("កក")` → `[0, 1, 2]` (two clusters)

**Tests** (`src/__tests__/caret.test.ts`):

- Empty string → `[0]`
- Single consonant → `[0, 1]`
- Consonant + coeng + subscript → `[0, clusterLength]` (single cluster)
- Multi-cluster text → boundaries at each cluster edge
- Mixed Khmer + Latin → cluster-safe for Khmer, character-safe for Latin
- Normalized vs raw text behavior
- Round-trip: positions cover the entire string with no gaps

### 2.2 `deleteBackward(text, cursorIndex, options?)`

**File**: `src/core/caret.ts` (same file)

Deletes the cluster (or character) before the cursor, respecting cluster boundaries. Uses `getCaretBoundaries()` internally to find the previous safe position.

```ts
interface DeleteResult {
    text: string;
    cursorIndex: number;
}

function deleteBackward(
    text: string,
    cursorIndex: number,
    options?: CaretOptions
): DeleteResult;
```

**Behavior**:

- If cursor is at position 0, returns `{ text, cursorIndex: 0 }` unchanged.
- `cursorIndex` validation:
    - If `cursorIndex < 0`, clamp to `0`.
    - If `cursorIndex > text.length`, clamp to `text.length`.
    - If `cursorIndex` is not an integer, throw `TypeError`.
- Finds the previous caret boundary before `cursorIndex`.
- Deletes the text between that boundary and the cursor.
- Returns the modified text and the new cursor position (now at the previous boundary).
- Default behavior (`normalize: false`) keeps all indices in original input space.
- If `normalize: true` is used, the operation occurs in normalized-text space and the returned `text` / `cursorIndex` are normalized-space outputs.

**Examples**:

- `deleteBackward("កក", 2)` → `{ text: "ក", cursorIndex: 1 }` — deletes last cluster
- `deleteBackward("ក្កក", 3)` → `{ text: "ក្ក", cursorIndex: 2 }` — deletes last cluster
- `deleteBackward("ក", 0)` → `{ text: "ក", cursorIndex: 0 }` — no-op at start
- `deleteBackward("កក", 1)` → `{ text: "ក", cursorIndex: 0 }` — deletes first cluster
- `deleteBackward("កក", 999)` → same as end-of-string delete (cursor clamped to end)

**Tests** (in `src/__tests__/caret.test.ts`):

- Cursor at start → no-op
- Cursor at end → deletes last cluster
- Cursor in middle → deletes preceding cluster
- Cluster with coeng + subscript → deletes entire visual unit
- Mixed Khmer + Latin → correct per-character-type behavior
- Empty string → no-op
- Consecutive deletes (calling twice) → consistent state
- Cursor index validation (negative / too large / non-integer)

## Open Design Decisions

1. **`getCaretBoundaries` return type**: Just indices (`number[]`) vs rich objects. Recommending just indices for simplicity — callers can intersect with `getClusterBoundaries()` for richer data.

2. **`deleteBackward` normalization**: Operates on raw text by default (`normalize: false`) so cursor semantics are stable for editor use-cases. When `normalize: true` is passed, operation and outputs are in normalized-text space.

3. **Normalization option**: Both APIs accept `normalize?: boolean` with default `false` to preserve caller index space. When enabled, positions/results are relative to normalized text.
