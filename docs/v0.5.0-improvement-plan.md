# v0.5.0 Improvement Plan (Current Version Hardening)

This plan focuses on improving reliability, maintainability, and release confidence for the current implementation before adding major new features.

## Objectives

- Prevent silent behavior mismatches in public APIs.
- Reduce algorithm drift between segmentation components.
- Make performance/accuracy checks reliable in CI.
- Strengthen code-quality guardrails.
- Clarify release/runtime compatibility and public API surface.

## Priority Overview

| Priority | Theme                         | Why it matters                                          | Target outcome                                   |
| -------- | ----------------------------- | ------------------------------------------------------- | ------------------------------------------------ |
| P0       | Runtime option validation     | Avoid silent wrong behavior in production               | Invalid options fail fast with clear errors      |
| P1       | Viterbi/cluster consolidation | Reduce correctness drift and long-term maintenance cost | Shared cluster traversal model across algorithms |
| P1       | CI performance reliability    | Prevent unnoticed regressions and flaky confidence      | Stable, required perf/accuracy gate              |
| P2       | Lint and safety rules         | Catch risky patterns earlier                            | Consistent static checks in local + CI           |
| P2       | Packaging/API clarity         | Improve user experience and release predictability      | Explicit Node support + clear exports            |

## Workstreams

### Workstream A (P0): Runtime Validation for Public Options

**Scope**

- Validate `strategy` and other externally provided options at runtime.
- Replace silent fallback behavior with explicit failure semantics.

**Tasks**

- [ ] Add a dedicated options validator in `src/core/segment.ts` (or a new shared helper).
- [ ] Throw `TypeError` with actionable message for invalid `strategy`.
- [ ] Add tests for:
    - [ ] valid strategies (`fmm`, `bmm`, `bimm`, `viterbi`)
    - [ ] invalid string values
    - [ ] non-string values from untyped callers
- [ ] Document runtime validation behavior in `README.md`.

**Acceptance Criteria**

- Invalid strategy values never silently change algorithm behavior.
- Error message includes the invalid value and allowed values.
- New tests pass in `npm test`.

**Estimate**: 0.5-1 day

---

### Workstream B (P1): Consolidate Viterbi and Cluster Boundary Logic

**Scope**

- Remove duplicate Khmer cluster traversal logic between:
    - `src/core/cluster.ts`
    - `src/algorithms/viterbi.ts`
- Improve Viterbi internals to avoid avoidable string-concatenation overhead.

**Tasks**

- [ ] Design a shared cluster-walking utility (single source of truth for cluster boundaries).
- [ ] Refactor `viterbi.ts` to operate on cluster indices instead of re-deriving character boundaries.
- [ ] Replace repeated candidate string concatenation with a lower-allocation approach.
- [ ] Add regression tests for edge cases:
    - [ ] Robat and coeng/subscript sequences
    - [ ] mixed Khmer + digits + separators
    - [ ] unknown-word spans
- [ ] Re-run benchmark comparisons against current baseline.

**Acceptance Criteria**

- Viterbi and core cluster logic agree on boundaries for shared test fixtures.
- No regression in segmentation outputs on existing test corpus.
- Runtime does not exceed current p95 by more than agreed tolerance.

**Estimate**: 3-5 days

---

### Workstream C (P1): CI Performance/Accuracy Gate Reliability

**Scope**

- Convert performance checks from best-effort to dependable signal.
- Reduce false positives/negatives due to environment variance.

**Tasks**

- [ ] Remove `continue-on-error` for performance checks once baseline is stable.
- [ ] Replace fixed wall-clock assertions in perf tests with baseline-relative checks.
- [ ] Add warmup/repeat strategy for perf runs and summarize median/p95.
- [ ] Ensure benchmark dataset availability is deterministic in CI:
    - [ ] cache artifact, or
    - [ ] commit a minimal deterministic sample set for gate checks.
- [ ] Document perf gate policy and update `docs/benchmark-methodology.md`.

**Acceptance Criteria**

- CI fails on meaningful regressions, not random runner jitter.
- Perf/accuracy jobs are reproducible across two consecutive CI runs.
- Team can explain pass/fail outcomes from generated metrics.

**Estimate**: 2-4 days

---

### Workstream D (P2): Linting and Type-Safety Guardrails

**Scope**

- Expand static quality checks beyond TypeScript compile-only linting.

**Tasks**

- [ ] Add ESLint (or Biome) configuration with high-signal rules.
- [ ] Integrate lint into CI and local scripts.
- [ ] Reduce risky non-null assertions in hot code paths:
    - [ ] `src/core/cluster.ts`
    - [ ] `src/algorithms/viterbi.ts`
    - [ ] `src/core/detect.ts`
- [ ] Add targeted tests where nullability handling changes behavior.

**Acceptance Criteria**

- Lint runs in CI and blocks on rule violations.
- Critical parsing paths avoid unsafe non-null assertions where feasible.
- No behavior regressions in existing tests.

**Estimate**: 1-2 days

---

### Workstream E (P2): Packaging and Public API Polish

**Scope**

- Improve compatibility signaling and API discoverability.

**Tasks**

- [ ] Add `engines.node` in `package.json` aligned with actual runtime/script features.
- [ ] Review exports in `src/index.ts` and public types in `src/types/public.ts`.
- [ ] Remove or wire any exported types not supported by stable public APIs.
- [ ] Ensure README API section exactly matches exported public surface.

**Acceptance Criteria**

- Consumers see clear Node compatibility in package metadata.
- Public API docs and exports are consistent.
- No ambiguous/unused exported types in published entrypoint.

**Estimate**: 0.5-1 day

## Execution Order

1. Workstream A (fast risk reduction)
2. Workstream C (establish trustworthy CI signal)
3. Workstream B (larger refactor with safety net from A + C)
4. Workstream D
5. Workstream E
6. Final full validation + release prep

## Validation Checklist (Release Gate)

- [ ] `npm run build`
- [ ] `npm test`
- [ ] `npm run lint`
- [ ] `npm run format:check`
- [ ] `npm run test:perf`
- [ ] `npm run test:accuracy`

## Risks and Mitigations

- **Risk:** Refactor in Viterbi changes segmentation behavior subtly.  
  **Mitigation:** Add golden fixtures + benchmark diff check before merge.

- **Risk:** Perf gate becomes noisy and blocks development.  
  **Mitigation:** Use baseline-relative metrics with repeat runs and tolerance windows.

- **Risk:** Lint rollout creates high initial churn.  
  **Mitigation:** Start with a small ruleset, escalate strictness gradually.

## Definition of Done (v0.5.0 Plan)

- P0/P1 workstreams complete and merged.
- CI gates are required and stable.
- Benchmarks show no unacceptable regression versus baseline.
- Public API/runtime expectations are explicit in package metadata and docs.
